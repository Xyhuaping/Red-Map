## 1. 功能概述

### 1.1 功能定义

心跳查询功能是指 APP 在运行期间自动、周期性地跟踪用户当前位置，当系统检测到用户进入预设的电子围栏区域时，自动触发弹窗提醒，引导用户与围栏关联的红色人物进行 AR 交互和 AI 对话。

### 1.2 核心需求

| 编号 | 需求项 | 描述 |
|------|--------|------|
| HB-01 | 自动位置跟踪 | APP 自动获取用户实时位置，无需手动触发 |
| HB-02 | 围栏触发检测 | 当用户进入电子围栏区域时自动识别并触发 |
| HB-03 | 弹窗提醒 | 前台时 APP 内弹窗，后台时系统通知推送 |
| HB-04 | 冷却机制 | 同一围栏在冷却时间内不重复触发 |
| HB-05 | 隐私合规 | 遵循位置权限授权流程，明确告知用户用途 |

### 1.3 用户故事

1. 作为游客，我希望打开 APP 后自动检测附近红色文化景点，以便及时发现并参观
2. 作为游客，我希望进入景点区域时收到弹窗提醒，以便了解该处关联的红色人物
3. 作为游客，我希望弹窗能直接引导我进入 AR 游览或 AI 对话，以便快速体验
4. 作为游客，我希望短时间内不会重复收到同一景点的提醒，以免被打扰
5. 作为游客，我希望 APP 在后台运行时也能收到围栏提醒通知，以便不错过景点
6. 作为管理员，我希望能在后台配置围栏的触发参数，以便灵活调整运营策略
7. 作为管理员，我希望查看围栏触发统计数据，以便分析用户行为和优化围栏布局

### 1.4 确定的技术决策

| 决策项 | 选定方案 | 理由 |
|--------|----------|------|
| 定位策略 | JS 层心跳轮询 + 后端围栏检测 | 复用现有 `location.js` / `fence.js` / `/fences/check`，与项目纯 JS 调性一致，无需编写原生 Java 插件 |
| 弹窗方式 | APP内弹窗 + 系统通知结合 | 前台弹窗交互丰富，后台通知不遗漏 |
| 心跳间隔 | 15秒 | 平衡实时性与耗电，步行速度下延迟约15米 |
| 冷却时间 | 5分钟 | 避免频繁打扰，适合一般游览场景 |
| 通知方案 | `@capacitor/local-notifications` 官方插件 | 无需自定义原生代码，Capacitor 官方维护，与项目零原生代码理念一致 |

### 3.2 JS 层心跳服务（核心）

#### 3.2.1 设计理念

心跳服务的核心思路是**完全复用项目已有的前端定位和围栏检测能力**，通过 JS 层定时轮询实现，不引入任何原生 Java 代码：

- **定位**：复用 `location.js` 的 `LocationService.getCurrentLocation()`（三级降级：高德→浏览器→默认）
- **围栏预检**：复用 `fence.js` 的 `FenceService.findTriggeredFences()`（Haversine + 射线法，纯前端计算）
- **后端验证**：复用 `fences.js` 的 `FencesService.checkFence()`（调用 `POST /fences/check`）
- **状态管理**：复用 `store/index.js` 的 `mapStore`（`addTriggeredFence` / `setLocation`）

#### 3.2.2 HeartbeatService 类设计

```javascript
// js/services/heartbeat.js (新建)
import locationService from '../utils/location.js'
import fenceService from '../utils/fence.js'
import fencesService from './fences.js'
import { mapStore, userStore } from '../store/index.js'
import storage from '../utils/storage.js'
import { throttle } from '../utils/helpers.js'

class HeartbeatService {
    constructor() {
        this.isRunning = false
        this.timerId = null
        this.intervalMs = 15000             // 心跳间隔 15秒
        this.cooldownMs = 5 * 60 * 1000     // 冷却时间 5分钟
        this.syncIntervalMs = 30 * 60 * 1000 // 围栏同步间隔 30分钟
        this.syncTimerId = null
        this.cachedFences = []               // 本地缓存的活跃围栏
        this.triggerHistory = new Map()      // fenceId → lastTriggerTime
        this.popup = null                    // FencePopup 实例
        this._loadTriggerHistory()           // 从 localStorage 恢复冷却
    }

    async start()                // 启动心跳服务
    async stop()                 // 停止心跳服务
    async syncFences()           // 从后端同步活跃围栏到本地缓存
    async _heartbeatTick()       // 单次心跳：定位 → 预检 → 后端验证 → 触发处理
    _checkCooldown(fenceId)      // 冷却检查
    _recordTrigger(fenceId)      // 记录触发时间（内存 + localStorage）
    _loadTriggerHistory()        // 启动时从 localStorage 恢复冷却历史
    _handleTrigger(fence, figure)// 统一触发处理：前后台判断 → 弹窗/通知
    _showFencePopup(fence, figure)   // 前台弹窗
    _showNotification(fence, figure) // 后台通知
    _isAppForeground()           // 判断 APP 前后台状态
    _recordTriggerToBackend(fenceId, figureId) // 上报触发记录到后端
}
```


#### 3.2.4 关键实现细节

**定位获取**：复用现有 `location.js`，无需修改：

```javascript
async _heartbeatTick() {
    try {
        const location = await locationService.getCurrentLocation({
            enableHighAccuracy: true,
            timeout: 10000,
            maximumAge: 5000
        })
        if (!location || !location.lng) return

        mapStore.setLocation(location)

        const point = { lng: location.lng, lat: location.lat }
        const triggered = fenceService.findTriggeredFences(point, this.cachedFences)
        if (triggered.length === 0) return

        const result = await fencesService.checkFence(point.lng, point.lat)
        if (!result.triggered) return

        if (!this._checkCooldown(result.fence.id)) return

        this._handleTrigger(result.fence, result.figure)
    } catch (e) {
        console.warn('[Heartbeat] tick failed:', e)
    }
}
```

**冷却机制**：

```javascript
_checkCooldown(fenceId) {
    const lastTime = this.triggerHistory.get(fenceId)
    if (!lastTime) return true
    return Date.now() - lastTime >= this.cooldownMs
}

_recordTrigger(fenceId) {
    this.triggerHistory.set(fenceId, Date.now())
    const history = Object.fromEntries(this.triggerHistory)
    storage.set('trigger_history', history)
}

_loadTriggerHistory() {
    const history = storage.get('trigger_history', {})
    for (const [fenceId, timestamp] of Object.entries(history)) {
        if (Date.now() - timestamp < this.cooldownMs) {
            this.triggerHistory.set(fenceId, timestamp)
        }
    }
}
```

**围栏缓存与同步**：

```javascript
async syncFences() {
    try {
        const response = await fencesService.getFences({ is_active: true })
        this.cachedFences = response.data || []
        storage.set('cached_fences', this.cachedFences)
    } catch (e) {
        this.cachedFences = storage.get('cached_fences', [])
    }
}
```

**前后台判断**：

```javascript
_isAppForeground() {
    return document.visibilityState === 'visible'
}
```

### 3.3 本地通知（`@capacitor/local-notifications`）

#### 3.3.1 方案选择

使用 Capacitor 官方维护的 `@capacitor/local-notifications` 插件，无需编写任何原生 Java 代码：

| 对比项 | 官方插件 | 自定义原生插件 |
|--------|----------|----------------|
| 原生代码量 | 0 | ~200行 Java |
| 维护成本 | 官方维护 | 自行维护 |
| 与项目调性 | 一致（纯 JS 调用） | 不一致（需写 Java） |
| 功能 | 本地通知（满足需求） | 可深度定制 |
| 安装方式 | `npm install` | 手动创建 Java 类 |

#### 3.3.2 集成方式

```bash
npm install @capacitor/local-notifications
npx cap sync android
```

#### 3.3.3 通知发送

```javascript
import { LocalNotifications } from '@capacitor/local-notifications'

async _showNotification(fence, figure) {
    const hasPermission = await LocalNotifications.checkPermissions()
    if (hasPermission.display !== 'granted') return

    await LocalNotifications.schedule({
        notifications: [{
            title: `发现红色文化景点：${figure?.name || fence.name}`,
            body: fence.trigger_prompt || '您已进入景点区域，点击了解更多',
            id: fence.id,
            schedule: { at: new Date(Date.now() + 1000) },
            sound: undefined,
            attachments: undefined,
            actionTypeId: '',
            extra: {
                fenceId: fence.id,
                figureId: figure?.id,
                type: 'fence_trigger'
            }
        }]
    })
}
```

#### 3.4.1 FencePopup 设计

```javascript
// js/components/FencePopup.js (新建)
class FencePopup {
    constructor(container) {
        this.container = container

    show(fenceData) {
        // fenceData: { fence, figure }
```

### 3.6 权限授权流程

#### 3.6.1 授权时序

```
APP首次启动
  │
  ├── 弹出"位置权限说明"弹窗
  │   "我们需要获取您的位置，以便在您靠近红色文化景点时及时提醒"
  │   [同意] → 请求 ACCESS_FINE_LOCATION + ACCESS_COARSE_LOCATION
  │   [拒绝] → 进入游客模式，无围栏提醒
  │
  ├── Android 13+ 通知权限
  │   弹出"通知权限说明"弹窗
  │   "允许通知，以便在后台时向您推送景点提醒"
  │   [允许] → 请求 POST_NOTIFICATIONS
  │   [拒绝] → 仅前台弹窗
  │
  └── 权限配置完成 → 启动心跳服务
```

> ⚠️ **【需确认】** 权限说明弹窗的文案内容需与产品经理确认  
> ⚠️ **【需确认】** 用户拒绝权限后的降级策略需与产品经理确认

#### 3.6.2 降级策略

| 权限状态 | 可用功能 |
|----------|----------|
| 全部授予 | 前台围栏检测 + 弹窗 + 后台通知 |
| 仅前台位置 | 前台围栏检测 + 弹窗，无后台通知 |
| 仅粗略位置 | 围栏检测精度降低，可能遗漏小围栏 |
| 无位置权限 | 无围栏检测，手动浏览模式 |
| 无通知权限 | 前台弹窗正常，后台无通知 |

#### 3.6.3 与原方案的区别

本方案**不需要** `ACCESS_BACKGROUND_LOCATION` 和 `FOREGROUND_SERVICE` 权限，因为：
- 心跳轮询仅在 APP WebView 运行时生效（前台或后台未冻结时）
- 不依赖 Android 原生地理围栏，无需后台持续定位
- 后台通知依赖 Capacitor 官方插件，而非自定义前台服务

**M1 关键实现细节**：

- 定时器使用 `setTimeout` 链式调用（而非 `setInterval`），确保上一次心跳完成后才开始下一次计时，避免并发问题
- 前端预检使用 `fence.js` 的 `findTriggeredFences()`，仅当本地检测到触发时才调用后端 API，减少网络请求
- 围栏缓存使用 `localStorage`，APP 重启后可立即使用缓存数据，无需等待首次同步
- 冷却历史从 `localStorage` 恢复时，自动清理已过期的冷却记录
- 心跳在 `document.visibilitychange` 事件中可动态调整：后台时暂停心跳（节省资源），前台时恢复

> ⚠️ **【需确认】** APP 进入后台后心跳是否暂停？  
> 方案A：暂停心跳，仅依赖通知插件的后台能力（推荐，省电）  
> 方案B：降低频率继续心跳（如60秒间隔），但 WebView 可能被系统冻结

## 7. 风险评估

### 7.1 技术风险

| 风险 | 等级 | 描述 | 应对措施 |
|------|------|------|----------|
| 后台 WebView 冻结 | 高 | Android 系统可能在 APP 后台时冻结 WebView，导致心跳停止 | APP 后台时暂停心跳，依赖 LocalNotifications 的定时通知能力；前台恢复后立即 tick |
| 定位精度不足 | 中 | 高德/浏览器定位在室内或信号弱时精度低 | 检测定位精度，精度低于阈值时暂缓围栏检测并提示用户 |
| 前端预检与后端结果不一致 | 中 | 浮点精度差异可能导致前端预检触发但后端验证未触发 | 以前端预检为初筛，以后端验证为最终结果，不一致时静默忽略 |
| 围栏缓存过期 | 低 | 30分钟同步间隔内围栏数据可能变更 | 缓存有效期标记，过期后强制同步；管理员操作围栏后可触发推送更新 |
| LocalNotifications 兼容性 | 低 | 部分厂商 ROM 可能限制本地通知 | 测试主流机型，降级为仅前台弹窗 |

### 7.2 产品风险

| 风险 | 等级 | 描述 | 应对措施 |
|------|------|------|----------|
| 用户拒绝位置权限 | 高 | 隐私意识强的用户可能拒绝 | 清晰的权限说明文案，降级使用模式 |
| 频繁弹窗打扰用户 | 中 | 围栏密集区域可能频繁触发 | 冷却机制 + 用户可关闭心跳功能 |
| 耗电投诉 | 中 | 15秒定位轮询导致耗电增加 | 前端预检减少网络请求；后台暂停心跳；提供设置开关 |
| 误触发 | 中 | 路过围栏但未停留时触发 | 可考虑增加停留检测（连续2-3次心跳均在围栏内才触发） |
| 后台无提醒 | 中 | APP 后台时心跳暂停，无法检测围栏 | 通过通知引导用户保持 APP 前台；后续版本评估原生围栏方案 |

### 7.3 合规风险

| 风险 | 等级 | 描述 | 应对措施 |
|------|------|------|----------|
| 位置数据隐私 | 高 | 收集用户位置数据需合规 | 仅在用户授权后收集，明确告知用途，提供关闭选项 |
| 数据存储合规 | 中 | 位置数据存储需符合个人信息保护法 | 最小化存储，定期清理，匿名化处理 |

> ⚠️ **【需确认】** 位置数据是否需要上传到服务器？还是仅在本地处理？  
> ⚠️ **【需确认】** 是否需要隐私政策更新以涵盖位置数据收集？

### E. 方案演进路径

当前 JS 层心跳轮询方案是**第一阶段**的务实选择，后续可根据实际需求演进：

```
第一阶段（当前）           第二阶段（可选）            第三阶段（可选）
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ JS心跳轮询    │  →   │ + 原生围栏    │  →   │ + 后台服务    │
│ + 前端预检    │      │ (圆形围栏)   │      │ (持续定位)   │
│ + 后端验证    │      │ + 后端验证    │      │ + 后端验证    │
│ + 前台弹窗    │      │ + 前后台通知  │      │ + 前后台通知  │
│ + 后台通知    │      │              │      │              │
│ (仅前台检测)  │      │ (前后台检测)  │      │ (持续检测)   │
└──────────────┘      └──────────────┘      └──────────────┘
  工期: 18天             工期: +5天             工期: +3天
  原生代码: 0行          原生代码: ~200行       原生代码: ~300行
```

第二阶段在第一阶段基础上，仅新增 Android 原生 `GeofencePlugin` 处理圆形围栏的后台检测，多边形围栏仍由 JS 层处理。第三阶段新增 `ForegroundService` 实现持续后台定位。
