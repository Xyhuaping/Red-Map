## 二、重构目标与范围

### 2.1 核心目标

1. **安全加固**：修复全部安全隐患，建立完善的认证授权体系
2. **架构升级**：引入分层架构，实现业务逻辑与路由解耦
3. **功能完善**：实现后台管理系统，包含围栏可视化绘制功能
4. **前端兼容**：确保与前端改造计划（纯 HTML/CSS/JS + Fetch API）无缝对接
5. **生产就绪**：完善日志、监控、限流、异常处理等运维基础设施
6. **Ubuntu 适配**：确保系统在 Ubuntu Linux 环境下稳定运行

### 2.2 重构范围

| 范围 | 说明 | 改造程度 |
|------|------|---------|
| 认证授权 | 从 SHA256 升级为 bcrypt，增加 RBAC 权限体系 | 🔴 重构 |
| API 层 | 保持接口路径兼容，增加版本前缀和分页元数据 | 🟡 改造 |
| 业务逻辑 | 抽取 Service 层，路由层仅负责参数校验和响应 | 🔴 重构 |
| 数据模型 | 增加角色字段、软删除、多边形围栏支持 | 🟡 扩展 |
| 数据库迁移 | 引入 Alembic 管理表结构变更 | 🟢 新增 |
| 后台管理 | 全新开发 Web 管理后台 | 🔴 新增 |
| 围栏可视化 | 集成高德地图实现围栏绘制与管理 | 🔴 新增 |
| 文件管理 | 实现文件上传、存储、访问的完整方案 | 🟢 新增 |
| WebSocket | AI 对话支持流式输出 | 🟢 新增 |
| 运维基础设施 | 日志、限流、健康检查、异常处理 | 🟢 新增 |

### 2.3 兼容性保障原则

- **API 路径兼容**：现有 `/api/*` 路径保持不变，新增接口使用 `/api/v1/*` 前缀
- **响应格式兼容**：现有响应结构不变，新增字段不影响前端解析
- **认证方式兼容**：继续使用 JWT Bearer Token，扩展 Token 中携带角色信息
- **数据库兼容**：表结构变更通过 Alembic 迁移脚本执行，不破坏现有数据

### 4.2 分层架构说明

| 层级 | 职责 | 对应目录 |
|------|------|---------|
| **路由层 (Router)** | 接收请求、参数校验、调用 Service、返回响应 | `app/api/` |
| **服务层 (Service)** | 业务逻辑编排、事务管理、调用 Repository | `app/services/` |
| **数据访问层 (Repository)** | 数据库 CRUD 操作、查询构建 | `app/repositories/` |
| **模型层 (Model)** | ORM 模型定义、数据库表映射 | `app/models/` |
| **模式层 (Schema)** | 请求/响应数据结构定义、数据校验 | `app/schemas/` |
| **核心层 (Core)** | 配置、安全、依赖注入、中间件 | `app/core/` |

```python
# app/core/security.py
import bcrypt

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return bcrypt.checkpw(
        plain_password.encode("utf-8"),
        hashed_password.encode("utf-8"),
    )

def get_password_hash(password: str) -> str:
    salt = bcrypt.gensalt()
    hashed = bcrypt.hashpw(password.encode("utf-8"), salt)
    return hashed.decode("utf-8")

def is_bcrypt_hash(hashed: str) -> bool:
    return hashed.startswith("$2b$") or hashed.startswith("$2a$")

def verify_legacy_password(plain_password: str, legacy_hash: str) -> bool:
    computed = hashlib.sha256(plain_password.encode("utf-8")).hexdigest()
    return computed == legacy_hash
```

```python
# app/core/dependencies.py
from fastapi import Depends, HTTPException, status
```

**密码迁移策略**：旧 SHA256 哈希在用户首次登录时自动升级为 bcrypt，无需强制重置密码。

```python
# app/services/auth.py (密码迁移逻辑)
def login(self, data: UserLogin) -> TokenResponse:
    user = self.user_repo.get_by_username(data.username)
    if not user:
        raise UnauthorizedException("用户名或密码错误")

    if not user.is_active:
        raise UnauthorizedException("账号已被禁用")

    if is_bcrypt_hash(user.password_hash):
        if not verify_password(data.password, user.password_hash):
            raise UnauthorizedException("用户名或密码错误")
    else:
        if not verify_legacy_password(data.password, user.password_hash):
            raise UnauthorizedException("用户名或密码错误")
        user.password_hash = get_password_hash(data.password)
        self.db.commit()
```

#### 5.3.2 围栏检测优化

**现有方案**：全表扫描 + Python Haversine 计算

**优化方案**：使用 MySQL 空间函数 `ST_Distance_Sphere`

```sql
SELECT *, ST_Distance_Sphere(
    POINT(center_lng, center_lat),
    POINT(#{lng}, #{lat})
) AS distance
FROM geo_fences
WHERE is_active = TRUE
  AND shape_type = 'circle'
HAVING distance <= radius
ORDER BY distance ASC
LIMIT 1;
```

**优化效果**：
- 减少 Python 层计算开销
- 利用 MySQL 空间索引加速查询
- 支持未来扩展到多边形围栏

#### 5.3.3 多边形围栏检测

对于多边形围栏，使用 MySQL 的 `ST_Contains` 函数：

```sql
SELECT * FROM geo_fences
WHERE is_active = TRUE
  AND shape_type = 'polygon'
  AND ST_Contains(
    ST_GeomFromText(polygon_wkt),
    POINT(#{lng}, #{lat})
  );
```

## 十一、风险评估与应对措施

### 11.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **密码哈希迁移失败** | 低 | 高 | 实现双哈希兼容验证，旧密码在首次登录时自动升级，无需强制重置；迁移前完整备份数据库 |
| **MySQL 空间函数性能不达预期** | 低 | 中 | 预先进行基准测试，准备 Python 回退方案；对 geo_fences 表建立空间索引 |
| **WebSocket 连接不稳定** | 中 | 中 | 实现断线重连机制，保留 HTTP 轮询作为降级方案；设置合理的心跳间隔 (30s) |
| **智谱 GLM API 限流/宕机** | 中 | 高 | 实现请求队列与重试机制 (指数退避)；准备本地缓存回复作为降级方案；对接多个 AI 服务商作为备选 |
| **Alembic 迁移冲突** | 中 | 中 | 制定迁移脚本开发规范，禁止手动修改数据库；多人协作时按顺序合并迁移脚本 |
| **内存限流重启丢失** | 低 | 低 | 限流计数器存储在进程内存，服务重启后重置；对于当前规模可接受，后续可按需引入 Redis |

### 11.2 业务风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **前端改造进度不匹配** | 中 | 高 | API 接口保持向后兼容，新旧接口并行运行；与前端团队建立每周同步机制 |
| **数据迁移丢失** | 低 | 高 | 迁移前完整备份；迁移脚本先在测试环境验证；生产迁移采用分步执行 + 数据校验 |
| **后台管理系统需求变更** | 中 | 中 | 采用模块化设计，功能模块可独立开发部署；预留扩展接口 |
| **用户密码迁移导致登录异常** | 中 | 高 | 实现透明迁移策略 (SHA256 → bcrypt 自动升级)；提供管理员手动重置密码功能；迁移期间开启登录日志监控 |

### 11.3 运维风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **Ubuntu 环境依赖冲突** | 低 | 中 | 使用 Python venv 隔离运行环境；Docker 化部署作为备选方案；锁定依赖版本 (requirements.txt) |
| **服务器资源不足** | 中 | 中 | 监控 CPU/内存/磁盘使用率；Nginx 开启 gzip 压缩；预留 30% 资源余量 |
| **日志磁盘占满** | 中 | 低 | Loguru 配置日志轮转 (50MB × 10 个文件)；定期清理过期日志；关键日志同时写入数据库 |
| **SSL 证书过期** | 低 | 中 | 使用 Certbot 自动续期；配置证书到期监控告警 |

### 11.4 安全风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **JWT Token 泄露** | 中 | 高 | 设置合理的 Token 过期时间 (7天)；实现 Token 黑名单（数据库表）；敏感操作要求二次验证 |
| **SQL 注入** | 低 | 高 | 全部使用 SQLAlchemy ORM 参数化查询；禁止拼接 SQL；输入校验使用 Pydantic |
| **文件上传漏洞** | 中 | 高 | 严格校验文件类型和大小；文件名随机化；上传目录禁止脚本执行；文件路径校验防止目录穿越 |
| **暴力破解** | 中 | 中 | 登录接口限流 (5次/分钟/IP)；密码强度要求 (8位+字母+数字)；登录失败延迟响应 |
| **CORS 配置不当** | 低 | 高 | 生产环境严格限制 `allow_origins`；禁止使用通配符 `*`；仅允许已知域名 |

### 11.5 应急预案

| 场景 | 触发条件 | 应急操作 | 恢复时间目标 |
|------|---------|---------|-------------|
| **数据库宕机** | MySQL 不可连接 | 1. 检查 MySQL 进程状态 2. 重启 MySQL 服务 3. 检查磁盘空间 4. 必要时从备份恢复 | ≤ 30 分钟 |
| **API 服务崩溃** | Uvicorn 进程退出 | 1. Supervisor 自动重启 2. 检查错误日志 3. 必要时回滚到上一版本 | ≤ 5 分钟 |
| **磁盘空间不足** | 使用率 > 90% | 1. 清理过期日志 2. 清理临时文件 3. 扩展磁盘容量 | ≤ 15 分钟 |
| **AI 服务不可用** | 智谱 API 连续失败 | 1. 返回缓存回复 2. 提示用户稍后重试 3. 切换备选 AI 服务 | ≤ 5 分钟 |
